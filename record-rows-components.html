Record Rows

Record rows are designed for repeated, scan-first content. They prioritize clarity, consistency, and density over visual emphasis.

Basic Record Row

The base record row presents a single item with a clear title and supporting metadata. It is designed for fast scanning and consistent alignment across repeated data.

Use this pattern when content is straightforward and does not require status indicators, selection, or extended detail.

Selectable Record Rows

Record rows can support selection when users need to choose one or more items from a list. Selection controls appear at the start of the row and do not change the overall layout or hierarchy.

Use this pattern for workflows such as including assessments, assigning items, or bulk actions. Selection should remain lightweight and not introduce additional container styles.

Record Row Overview

Record rows are compact, high-density containers for repeatable records inside panels. Use them for services, assessments, goals, and similar entities when the interface needs strong scanability, explicit selection, and lightweight actions without the weight of a full card.

Anatomy

The row keeps a compact hierarchy: selection first, then record identity, then optional actions. The top row may also contain a distinct status treatment with a divider, dot, and heavier label when row state needs to remain separate from the badge.

Parts

  • Selection control: optional; supports explicit record selection.
  • Eyebrow: optional; identifies the record type such as Service, Assessment, or Goal.
  • Badge: optional; surfaces a short status or category.
  • Title: required; primary record identifier.
  • Metadata lines: one or two compact lines of supporting detail using label/value pairs when needed.
  • Top-row status: optional; uses a divider, compact tag, small dot, and state color without replacing the badge.
  • Row actions: optional; typically edit and delete, revealed on hover or focus.

Variants and Optional Elements

Record rows should stay compact. Variations come from the record content, badge treatment, and whether selection or actions are needed.

Supported Patterns

  • Basic record row
  • Selectable record row
  • Service row
  • Assessment row
  • Goal row

Optional Elements

  • Selection checkbox
  • Eyebrow
  • Status or category badge
  • Metadata status treatment
  • Row actions
  • Second metadata line

Specifications

These rules define the intended behavior and structure for dense record listings inside panels.

Structure

  • Use record rows inside a larger parent container such as a panel or section.
  • Each row should keep title and metadata in a single compact column.
  • Selection sits on the far left, content in the center, and actions align right within the top row.
  • Use label/value metadata for dates, providers, examiners, and similar structured details.
  • When a row needs both category and state, keep both in the top row and separate them with a thin divider.
  • Use dividers between rows instead of independent card borders.

Sizing

  • Rows use compact vertical padding with a 28px left inset for content and rail spacing.
  • Title typography is 14px at 600 weight.
  • Metadata lines use 12.5px type for dense scanning.
  • Badges retain the same compact 20px pill height used by cards.

Behavior

  • The row itself may toggle selection, but it must not behave like an edit button.
  • Hover reveals the internal accent rail and row actions.
  • Checked rows use a subtle selected background without a tinted hover fill.
  • State rows may use text treatments that match the status color family.
  • Edit and delete remain explicit icon actions with distinct hover colors.

Accessibility

  • Selection uses a real checkbox so state remains keyboard accessible.
  • Action controls are explicit buttons with visible focus states.
  • Hover-only affordances should also appear on focus within the row.
  • Do not rely on the row title alone to communicate selection or action state.
