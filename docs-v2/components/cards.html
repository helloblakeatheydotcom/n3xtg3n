Card Overview

Cards are lightweight content containers used for grouped summaries, overview information, and low-interaction content. They are useful when information needs structure and separation, but does not require the stronger sectional hierarchy of a panel.

Parts

Container

Defines the card boundary, spacing, and lightweight surface treatment.

Eyebrow/label

Optional; provides small contextual framing above the title.

Title

Optional but common; names the content inside the card.

Body

Common; contains the primary summary or content.

Metadata row

Optional; supports compact secondary information such as date, status, or category.

Header actions

Optional; edit and delete icon buttons stay explicit and remain visible at all times.

Footer actions

Optional; provides lightweight next-step actions.

Variants and Optional Elements

Supported Patterns

  • Basic card
  • Card with metadata
  • Card with actions
  • Card in a grid

Optional Elements

  • Eyebrow
  • Metadata row
  • Footer actions
  • Status or badge
  • Supporting list
  • Edit and delete actions

Specifications

Structure

  • Cards are built from a single lightweight container.
  • Title and body are common, but all internal regions are optional.
  • Metadata should use a single inline pattern with separators between items.
  • When metadata follows body content, separate it with a divider and top padding.
  • Footer actions should remain lightweight and limited in number.

Layout

  • Cards use consistent internal padding and vertical spacing.
  • Cards should work equally well as stand-alone summaries or in grids.
  • Action icons align to the right of the title row without turning the full card into a click target.
  • Metadata and footer actions should read as secondary to the main content.

Behavior

  • Cards are static by default.
  • Hover reveals the internal accent rail without tinting the card surface.
  • The full card should not behave like an edit button; actions remain explicit.

Accessibility

  • Use semantic headings to match the surrounding page hierarchy.
  • Buttons and links inside cards must remain keyboard accessible.
  • Icon buttons need visible focus states and must remain reachable even when actions are visually quiet by default.
  • Status, metadata, and labels should maintain clear contrast and readable structure.
