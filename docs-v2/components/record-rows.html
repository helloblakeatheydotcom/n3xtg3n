Record Rows

Compact, scan-first rows for services, assessments, goals, and other repeatable records.

Basic, selectable, dense, and empty states Blue hover rail and selected state Anatomy, specs, and technical reference

Overview

Record rows are compact, high-density containers for repeatable records inside panels. Use them for services, assessments, goals, and similar entities when the interface needs strong scanability, explicit selection, and lightweight actions without the weight of a full card.

Variants

Use the examples below to compare the base row, badge and status treatment, selectable rows, empty containers, and denser row content.

Basic Record Row

The base record row presents a single item with a clear title and supporting metadata. It is designed for fast scanning and consistent alignment across repeated data.

Use this pattern when content is straightforward and does not require status indicators, selection, or extended detail.

Record Row With Badge/Status

Record rows can include both a badge and a distinct status when the row needs to communicate type and state at the same time. The badge should continue to identify the record category or delivery type.

Use a compact top-row status tag for states such as Waived or Inactive. Pair it with a small dot and a divider when row state needs to remain separate from the badge.

Selectable Record Rows

Record rows can support selection when users need to choose one or more items from a list. Selection controls appear at the start of the row and do not change the overall layout or hierarchy.

Use this pattern for workflows such as including assessments, assigning items, or bulk actions. Selection should remain lightweight and not introduce additional container styles.

Dense Record Row

Record rows can accommodate multiple lines of structured metadata while maintaining a compact, scan-friendly layout. Information should be grouped logically and presented in a consistent order across rows.

Use this pattern for more detailed items such as services, assessments, or plans where additional context is required, but a full card layout would add unnecessary visual weight.

Parts

Each numbered element corresponds to a part of the record row shown above.

Context

  1. Eyebrow Eyebrow: optional; identifies the record type such as Service, Assessment, or Goal.
  2. Status / Badge Badge: optional; surfaces a short status or category. Top-row status: optional; uses a divider, compact tag, small dot, and state color without replacing the badge.

Content

  1. Title Title: required; primary record identifier.
  2. Support Line Supporting line: optional; adds one compact line of context beneath the title.

Metadata

  1. Metadata Metadata lines: one or two compact lines of supporting detail using label/value pairs when needed.

Controls

  1. Actions Row actions: optional; typically edit and delete, and they remain visible at all times.

Layout and Behavior

These rules define how record rows behave within a container.

Layout

Record rows are used within a parent container such as a panel or section.

Content is arranged in a single compact column.

Selection aligns left, actions align right.

Content

Titles and supporting text remain grouped.

Supporting lines provide brief context beneath the title.

Metadata uses label/value pairs for structured details.

Status and Context

Category and state may both appear in the top row.

Separate them visually using spacing or a divider.

Do not replace the badge when displaying state.

Separation

Use dividers between rows.

Avoid individual card borders for each row.
