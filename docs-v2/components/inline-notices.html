Inline Notices

Contextual feedback patterns for info, warning, success, and error notices that live inside the current panel and support the task in progress.

Info, warning, success, error Titled and titleless variants Optional expandable details Anatomy, usage, token reference

Overview

Inline notices provide local, contextual feedback inside the current panel. They are meant to support the section the user is already working in, not interrupt or compete with page-level messaging.

Feedback Types

InfoContextual guidance that helps the user understand what belongs in the current section.

WarningIssues that need attention soon but do not fully block progress yet.

SuccessLocal confirmation that a supporting action completed successfully.

ErrorBlocking or corrective feedback that must be addressed inside the current panel.

Guidance

Use inline notices for local, section-specific feedback. Rules about placement, content, and semantic intent belong here rather than in the token reference.

Use Inline Notices When

  • The message is directly tied to the section the user is viewing.
  • The feedback should stay visually subordinate to the surrounding panel content.
  • The notice needs to explain, confirm, or warn without becoming a banner or global alert.
  • The message can remain concise and specific to the current workflow.
  • The notice belongs inside the panel body, not above the page shell.
  • Use expandable details only when the primary message should stay short but secondary guidance would still help.
  • Use info notices for contextual guidance that helps the user understand what belongs in the current section.
  • Use warning notices for issues that need attention soon but do not fully block progress.
  • Use success notices for local confirmation inside the current workflow, not for global save confirmation.
  • Use error notices only when the user must resolve an issue inside the current panel before continuing.
  • Omit the title only when the notice can stay clear and scannable as a single short sentence.

Anatomy

An inline notice contains a leading icon and supporting message content inside a lightly tinted container. Most notices use a short title plus body copy, and an optional disclosure may appear below the primary message when secondary details are helpful.

The icon sits on the left, aligned to the title row when a title is present. The optional disclosure sits beneath the primary message content and reveals secondary guidance without changing the default compact footprint.

Parts

Container

Lightly tinted surface with a 1px variant border and compact internal padding.

Icon

16px leading icon that communicates the notice type at a glance.

Title

Short directive or state label that anchors the message. This is optional when the notice can stay clear as one sentence.

Body text

Supportive instructional or confirmation copy that expands on the title.

Details toggle

Optional control that switches between hidden and revealed secondary guidance.

Details region

Optional supporting content area that appears below the primary notice copy when expanded.

Interaction

The disclosure pattern is optional, but when it is used its behavior should remain consistent and predictable.

  • The toggle label switches between “Show details” and “Hide details.”
  • The chevron rotates to indicate collapsed and expanded states.
  • The details region expands and collapses beneath the primary notice message.
  • A divider appears above the details region only when the notice is expanded.
  • The toggle updates aria-expanded and controls the associated details region through aria-controls.
  • Use the disclosure only when the primary message should stay concise and the secondary content is genuinely optional.
