Form Layout

Form Layout defines how fields are grouped inside panels for clear, accurate completion. This pattern applies the shared Layout foundation to form content, starting from a stacked field baseline and adding denser groupings only when the fields naturally support them.

Single-column-first Grouping by meaning Built on 12-column layout All fields live inside panels

Overview

A stacked field layout is the preferred baseline for most form flows. Add side-by-side fields only when they form a natural group and can still be completed clearly. The goal is accurate completion and readable structure, not maximum density.

Baseline Pattern

Start with stacked rowsMost form flows should read top to bottom so labels, helper text, and inputs remain easy to scan and complete.

Add columns only for natural groupsUse side-by-side placement when fields belong together semantically and users benefit from seeing them in the same row.

Prefer clarity over compactnessDo not increase density unless the field types, labels, and values remain readable at the chosen width.

Foundation Relationship

Built on the Layout foundationUse the shared 12-column Layout foundation as the base structure for gutters, containers, and spans.

Fields stay inside panelsApply the grid inside each panel body rather than placing loose fields directly on the page.

Completion order stays stableAs layouts expand or collapse, preserve a predictable reading and completion order rather than rearranging fields for visual balance alone.

Grid System

Form Layout uses the shared 12-column Layout foundation inside panel bodies. Depending on available width and field structure, the pattern can present content as one, two, or four visible groupings while preserving the same underlying grid.

Common Responsive Presentations

Apply the grid inside the panel body. See Foundations / Layout for the underlying 12-column base grid, 16px gutters, and field span system.

Context Grouping Guidance
Wider surfaces 4 columns Use only for short, structured sets when labels, values, and controls remain immediately clear.
Medium-width surfaces 2 columns Supports natural pairs and related groups without pushing controls too narrow.
Narrow surfaces 1 column Preserves the clearest top-to-bottom completion flow and remains the preferred baseline for most forms.

Field Placement Rules

Field spans and row groupings should support meaning, completion order, and readability. Use the grid to clarify relationships, not to make the layout feel tighter.

Column Span

  • Start standalone fields at span 4Use span 4 as the standard readable width for standalone inputs inside a panel-contained form layout.
  • Use span 6 for longer grouped fieldsPlace fields at span 6 when they need more room or when a paired grouping benefits from added width.
  • Use span 3 for compact structured setsReserve span 3 for short, structured inputs such as compact date parts, title and suffix, or other metadata-like fields.
  • Use span 12 for full-row contentExpand to the full row when labels, helper text, or control complexity require the widest available measure.

Row Grouping

  • Group only meaningful pairsPlace fields on the same row only when they form one conceptual unit, such as name parts, address fields, or linked dates.
  • Follow completion orderShared rows should support how users read and complete the form, not just how the layout can be packed.

Layout Patterns

Choose the pattern based on meaning, field length, and completion clarity. Do not increase density unless the group clearly benefits from it.

Stacked Layout

  • Preferred defaultUse stacked rows for most form flows so fields read in a straightforward top-to-bottom order.
  • Best for longer or support-heavy fieldsUse stacked placement when labels, helper text, or values need space to remain clear.

Paired Layout

  • Use for natural pairsUse paired rows when two fields belong together and benefit from being compared or completed side by side.
  • Keep the pairing obviousDo not create paired rows from unrelated fields just to shorten the page.

Dense Structured Layout

  • Use sparinglyReserve dense structured rows for short, structured sets where each field remains easy to scan and complete.
  • Back off quickly when fields growDo not keep dense groupings if labels wrap, controls narrow, or the row becomes harder to understand.

Grouping Strategies

Group fields by shared meaning first, then arrange them within the grid. Panel boundaries, row composition, and completion order should all reinforce the same structure.

Group by Shared Meaning

  • Keep meaningful groups intactNames, addresses, titles, and linked dates should read as one cluster rather than being split across unrelated rows.
  • Use panels to separate major groupsDo not rely on spacing alone to create major workflow boundaries. Panels should hold distinct related sets of content.

Preserve Completion Flow

  • Keep natural pairs on the same rowFields users mentally compare or complete together should stay aligned when width allows.
  • Do not optimize for symmetry aloneBalanced rows matter less than preserving meaning, sequence, and accurate completion.

Density Guidance

Choose density intentionally. Compact layouts are acceptable only when field types, labels, and values remain immediately understandable.

Increase Density Carefully

  • Increase density only for short structured contentCompact grouped fields work well when values are brief and the row is still easy to read at a glance.
  • Reduce density as soon as clarity dropsLong labels, helper text, or verbose values should trigger wider spans or fewer visible columns.

Protect Readability

  • Maintain stable rhythmKeep consistent gutters and panel body padding so dense groups still feel ordered.
  • Collapse earlyReduce visible columns before inputs become hard to read, scan, or interact with confidently.

Anti-Patterns

Avoid layout decisions that prioritize compactness over readability or that break the grouping logic users rely on while completing forms.

What to Avoid

  • Do not leave fields loose on the pageFields should live inside panels rather than floating directly in page layout.
  • Do not mix unrelated fields just to fill rowsShared rows should reflect shared meaning, not visual packing.
  • Do not use dense rows as the defaultStacked field layout should remain the starting point unless grouping clearly improves comprehension.
  • Do not force dense grouping for every setDense layouts are appropriate only for short, structured sets.
  • Do not invent one-off guttersUse the shared layout gutter instead of custom spacing values between fields.
